Sesión 21 — Seguridad en la Nube (Cloud Security)

🎭 METÁFORA

Subir a la nube sin entender el modelo de responsabilidad compartida es como mudarte a un edificio de departamentos y asumir que el seguro del edificio cubre tus muebles. El proveedor de nube asegura el edificio; tú aseguras todo lo que metes adentro. Muchas brechas en la nube ocurren porque el cliente no entendió su parte del contrato.

IaaS (Infrastructure as a Service)

El cliente gestiona: OS, aplicaciones, datos, red virtual. El proveedor: hardware físico, datacenter, hipervisor.

Ej: EC2, Azure VM

PaaS (Platform as a Service)

El cliente gestiona: aplicaciones y datos. El proveedor: OS, runtime, middleware, infraestructura.

Ej: AWS Elastic Beanstalk, Google App Engine

SaaS (Software as a Service)

El cliente gestiona: solo sus datos y accesos. El proveedor gestiona todo lo demás.

Ej: Office 365, Salesforce, Gmail

Modelo de Responsabilidad Compartida AWS

⚠️ RESPONSABILIDAD DEL CLIENTE ("security IN the cloud")
Datos de clientes · Cifrado datos en reposo y tránsito · IAM (identidades, contraseñas, MFA) · Configuración de OS · Firewall de red (Security Groups) · Aplicaciones web
🛡️ RESPONSABILIDAD DE AWS ("security OF the cloud")
Hardware físico · Datacenters · Hypervisor · Región/AZ · Servicios gestionados (S3, RDS, Lambda) · Red física global
📰 CASO REAL — Capital One Breach (2019)

Una ex-empleada de AWS explotó un misconfigured Web Application Firewall (WAF) y una función Lambda con exceso de privilegios para acceder a 100M de registros de clientes almacenados en S3. Costo: $80M en multas + daño reputacional masivo. El error fue del cliente (Capital One), no de AWS. El modelo de responsabilidad compartida fue clave en el litigio.

Vulnerabilidades Comunes en Entornos Cloud

Configuraciones Incorrectas (Misconfigurations)

  • S3 Bucket público: millones de registros expuestos sin intención
  • Security Groups abiertos: 0.0.0.0/0 en el puerto 22 (SSH)
  • IAM roles con permisos de administrador innecesarios (exceso privilegio)
  • CloudTrail deshabilitado: cero auditoría de actividad
  • MFA no habilitado en cuenta root de AWS

Ataques Específicos a Cloud

  • Cryptojacking: comprometer una instancia EC2 para minar criptomonedas a tu costo
  • Inyecciones en Lambda: funciones serverless sin validación de entrada
  • CI/CD pipeline attacks: comprometer el pipeline para inyectar código malicioso
  • Container escape: salir del contenedor al host Docker
  • Metadata API abuse: robar credenciales temporales desde la instancia
🔬 LABORATORIO AWS

Configurar: (1) IAM con grupos, roles y política de mínimo privilegio; (2) Bucket S3 con cifrado y bloqueo de acceso público; (3) Activar CloudTrail multi-región; (4) Revisar Trusted Advisor para recomendaciones de seguridad. Documentar hallazgos y correcciones.

Zero Trust en la Nube: "Never Trust, Always Verify"

🎭 METÁFORA

En el modelo tradicional, la red corporativa era el castillo y el perímetro era la muralla. Una vez adentro, confiabas en todos. En la nube no hay muralla. Zero Trust dice: tratar a TODOS los usuarios y sistemas como si vinieran de internet no confiable, sin importar si están "adentro" de la red.

Principios ZTA

  • Verificar siempre la identidad
  • Mínimo privilegio necesario
  • Asumir compromiso (assume breach)
  • Microsegmentación de la red
  • Monitoreo continuo

Implementación Práctica

  • MFA para TODOS los accesos
  • SSO + Identity Provider centralizado
  • Acceso condicional (device health check)
  • ZTNA reemplaza VPN tradicional
  • PAM para cuentas privilegiadas

Beneficios en la Nube

  • Elimina el problema del "insider confiable"
  • Reduce el blast radius de una cuenta comprometida
  • Compatible con trabajo remoto e híbrido
  • Auditoría granular de cada acceso

Sesión 22 — Ciberseguridad Industrial: OT, ICS y SCADA

🎭 METÁFORA

En IT, cuando un servidor falla, se reinicia. En OT, cuando un PLC falla, la línea de producción se para, el producto se echa a perder o hay riesgo de vida. La ciberseguridad industrial no puede ser igual a la de oficinas: aquí un parche mal aplicado puede causar un accidente físico.

OT (Operational Technology)

Sistemas que controlan o monitorean procesos físicos: manufactura, energía, agua, transporte. Prioridad: disponibilidad y seguridad física antes que confidencialidad.

ICS (Industrial Control Systems)

Categoría general de sistemas de control: SCADA, DCS, PLC, HMI. Diseñados para décadas de operación, no para actualizaciones frecuentes de seguridad.

SCADA

Supervisory Control and Data Acquisition: recopila datos de sensores remotos y permite control centralizado de infraestructura crítica (gasoductos, redes eléctricas, plantas).

Ataques Históricos a Infraestructura Crítica

📰 STUXNET (2010) — El primer arma cibernética

Diseñado por EE.UU. e Israel para sabotear el programa nuclear iraní. Infectó PLCs Siemens que controlaban centrifugadoras de uranio, causando que giraran fuera de control mientras mostraban lecturas normales a los operadores. Destruyó ~20% de las centrifugadoras sin dejar rastro aparente. Demostró que los sistemas OT eran vulnerables a ataques cibernéticos con consecuencias físicas.

Industroyer / CrashOverride (2016)

Malware que apagó la red eléctrica de Ucrania en diciembre. Afectó a 230,000 personas durante 6 horas. Primer malware diseñado específicamente para atacar sistemas de control de red eléctrica.

Colonial Pipeline (2021)

Ransomware DarkSide en la red IT (no OT) obligó a Colonial a apagar preventivamente el oleoducto. 5,700 km de oleoducto paralizado 6 días. Pagaron $4.4M de rescate. Escasez de combustible en EE.UU.

Tratamiento de Agua (Florida, 2021)

Atacante accedió remotamente al sistema de control de planta de agua potable y aumentó el nivel de soda cáustica 111 veces. Un operador lo detectó a tiempo. Vector: TeamViewer con contraseña compartida.

Modelo Purdue y Segmentación de Redes OT/IT

Nivel 5 — Red Corporativa (IT): ERP, email, internet, usuarios de negocio
Nivel 4 — Red de Planta (IT/OT Bridge): servidores de historia, bases de datos de producción, DMZ industrial
🔒 ZONA DESMILITARIZADA (DMZ) INDUSTRIAL — Firewall industrial + jump server + data diode
Nivel 3 — Red de Control: servidor SCADA, historians, servidores OPC
Nivel 2 — Red de Supervisión: HMI, estaciones de ingeniería
Nivel 1 — Red de Control de Campo: PLCs, RTUs, controladores de proceso
Nivel 0 — Proceso Físico: sensores, actuadores, motores, válvulas

La DMZ industrial impide comunicación directa entre IT y OT. El tráfico pasa por puntos de control auditados.

Estándares y Controles de Ciberseguridad Industrial

IEC 62443 — El Estándar Principal

  • Serie completa de normas para ciberseguridad de IACS (Industrial Automation and Control Systems)
  • Define niveles de seguridad (SL0-SL4) para zonas y conductos
  • Aplica a: fabricantes de equipos, integradores de sistemas, operadores de planta
  • IEC 62443-3-3: requisitos de sistema más implementado

NIST SP 800-82 Rev. 3

  • Guía para seguridad de sistemas de control industrial (ICS)
  • Cubre: riesgos, arquitecturas, controles de seguridad
  • Aplicado por: sector energético, manufactura, agua, transporte en EE.UU.
  • Compatible con NIST CSF
❓ QUIZ — Click para ver respuesta

Un PLC de 15 años en tu planta tiene una vulnerabilidad conocida (CVE publicado), pero el fabricante no ha lanzado un parche y la planta no puede parar. ¿Qué controles compensatorios implementas?

Controles compensatorios: (1) Segmentar el PLC en su propia VLAN con firewall que solo permite el tráfico estrictamente necesario; (2) Monitorear el tráfico hacia/desde el PLC (IDS industrial como Claroty, Dragos, Nozomi); (3) Eliminar todos los accesos remotos no esenciales al PLC; (4) Cambiar todas las credenciales por defecto; (5) Controlar físicamente el acceso al panel donde está el PLC; (6) Contactar al fabricante para ETA del parche o buscar alternativa de firmware; (7) Documentar el riesgo residual aceptado formalmente por la dirección.

Seguridad en IoT Industrial (IIoT)

Vulnerabilidades Típicas de IoT

  • Credenciales por defecto sin cambiar (admin/admin)
  • Firmware sin actualizar y sin mecanismo de update seguro
  • Comunicaciones sin cifrado (HTTP, Telnet)
  • Sin autenticación en API de gestión
  • Superficie de ataque amplia: Bluetooth, WiFi, Zigbee, LoRa
  • Dispositivos invisibles para el SIEM (sin agente)

Controles para IIoT

  • Inventario automatizado de todos los dispositivos IoT
  • Segmentación: VLAN dedicada por tipo de dispositivo
  • Cambiar credenciales por defecto en deploy
  • Actualización automática de firmware
  • Monitoreo pasivo de comportamiento (Claroty, Dragos)
  • Deshabilitar servicios y puertos no necesarios
  • Network Access Control (NAC) para validar antes de conectar
📰 CASO REAL — Mirai Botnet (2016)

Mirai infectó 600,000 dispositivos IoT (cámaras IP, routers, DVRs) usando credenciales por defecto. Los usó para lanzar el mayor DDoS registrado: 1.2 Tbps contra Dyn DNS, dejando sin acceso a Twitter, Netflix, Reddit y Amazon por horas. El vector: credenciales admin/admin que nadie había cambiado.

IA en Ciberseguridad: Doble Filo

IA como Defensa

  • Detección de anomalías: ML identifica comportamiento inusual de usuarios, procesos, red sin firmas predefinidas
  • UEBA (User Entity Behavior Analytics): detecta insider threats por desviación del baseline
  • Análisis predictivo: predice qué sistemas serán atacados basado en tendencias
  • Automatización SOC: SOAR + LLM para investigación automática de alertas
  • NLP: análisis automático de logs y threat intel en texto

IA en Manos del Atacante

  • Phishing con LLM: emails de spear phishing perfectamente redactados, sin errores gramaticales, personalizados con contexto OSINT
  • Deepfakes de audio/video: suplantar a CEO en llamada para autorizar transferencia
  • Malware polimórfico: código que muta automáticamente para evadir detección
  • Fuzzing automatizado: IA encuentra vulnerabilidades en software más rápido
  • Social engineering a escala: chatbots de phishing que mantienen conversaciones

Ciberseguridad en Vehículos Conectados

💡 DATO CLAVE

Un automóvil moderno tiene entre 70 y 150 ECUs (Electronic Control Units) y corre hasta 150 millones de líneas de código — más que un avión de combate. Con la conectividad V2X, 5G, actualizaciones OTA y sistemas ADAS, la superficie de ataque de un vehículo rivaliza con la de una empresa mediana.

Vectores de Ataque Vehicular

  • Bus CAN: sin autenticación, cualquier ECU puede enviar mensajes
  • OBD-II port: acceso físico total al bus
  • WiFi/Bluetooth del sistema infotainment
  • Carga inalámbrica y actualizaciones OTA
  • V2X (Vehicle-to-Everything): comunicaciones entre vehículos

Estándares Regulatorios

  • UNECE WP.29: obligatorio en UE, Japón, Corea para nuevos modelos desde 2022. Exige CSMS (Cybersecurity Management System) para fabricantes.
  • ISO/SAE 21434: estándar técnico de ciberseguridad en ingeniería de vehículos
  • TARA: Threat Analysis and Risk Assessment requerida para cada componente

Caso: Jeep Cherokee Hack (2015)

Miller y Valasek tomaron control remoto de un Jeep en movimiento a través del sistema Uconnect (WiFi/Sprint). Controlaron climatización, música, frenos y dirección. FCA retiró 1.4M de vehículos. El hack se publicó en Wired.

Seguridad en Transporte Inteligente e Industria 4.0

Flotas Conectadas y Telemática

  • Plataformas GPS/telemetría: superficie de ataque si no están protegidas
  • OBD-II dongles de terceros: acceso físico al bus CAN de toda la flota
  • APIs de las plataformas: autenticación robusta + rate limiting + WAF
  • Comunicación entre unidades y backend: TLS obligatorio, certificados válidos
  • Privacidad de datos de ubicación: LFPDPPP aplica a datos de conductores

Industria 4.0: Riesgos Emergentes

  • Digital Twin: gemelo digital puede usarse para planificar ataques físicos
  • Robots colaborativos (cobots): red interna accesible, firmware desactualizado
  • Manufactura aditiva (impresoras 3D): archivos STL modificados para sabotaje de calidad
  • Realidad aumentada para mantenimiento: dispositivos con WiFi en planta
  • Edge computing: procesamiento en planta sin los controles del datacenter central
🔬 ACTIVIDAD PRÁCTICA

Análisis de riesgo TARA para un sistema IoT industrial o vehículo conectado: identificar componentes, interfaces, amenazas por interfaz, impacto potencial, y proponer controles técnicos y organizativos para las amenazas de mayor riesgo.

Sesión 24 — Taller de Proyecto Integrador: Revisión y Preparación

🎭 METÁFORA

La sesión 24 es el ensayo general antes de la obra. Los actores ya saben sus líneas; ahora practican ante un público real, con las luces encendidas, para descubrir los momentos donde el ritmo se rompe o el argumento no convence. La retroalimentación del ensayo general es más valiosa que cualquier ensayo privado.

Estructura de la Sesión

  • 30 min por equipo: presentación de avance al grupo completo
  • Retroalimentación técnica del facilitador: rigor metodológico
  • Retroalimentación estratégica: ¿tiene sentido para el negocio?
  • Evaluación de pares: preguntas del grupo al equipo presentador
  • Ajustes identificados: lista de correcciones antes de S25

Checklist Previa a S25

  • Inventario de activos completo (ISO 27001)
  • Matriz de riesgos con ALE calculado para top riesgos
  • Plan de tratamiento con controles propuestos y costo
  • Política de seguridad redactada
  • Plan de respuesta a incidentes (NIST 800-61)
  • BCP con BIA y RTO/RPO definidos
  • Dashboard de métricas (Power BI/Excel)
  • Presentación ejecutiva: 15-20 slides máximo

¿Qué Busca el Panel Evaluador?

Comprensión del Negocio

¿Entiendes cómo la ciberseguridad impacta los objetivos de TU organización específica? No respuestas genéricas — contexto específico del sector.

Rigor Técnico

¿Tus análisis de riesgo tienen sustento (datos, benchmarks, metodología)? ¿Los controles propuestos son apropiados para los riesgos identificados?

Pensamiento Estratégico

¿Puedes defender por qué priorizas unos controles sobre otros? ¿Tienes en cuenta el presupuesto, la madurez actual y el ROI de seguridad?

Comunicación Ejecutiva

¿Tu presentación convencería a un CFO? ¿Usas lenguaje de riesgo financiero o solo jerga técnica? ¿Tienes recomendaciones accionables?

Manejo de Preguntas

¿Puedes responder preguntas fuera del guion? ¿Admites lo que no sabes? ¿Manejas la presión sin perder el hilo?

Completud del Entregable

¿Todos los entregables requeridos están presentes, completos y coherentes entre sí? ¿Hay un hilo conductor claro entre el análisis y las recomendaciones?

Cómo Construir una Narrativa Ejecutiva Ganadora

🎭 METÁFORA

La diferencia entre un informe técnico y una presentación ejecutiva es la misma que entre un diagnóstico médico lleno de términos técnicos y la explicación del médico al paciente: mismo problema, dos lenguajes completamente distintos. El panel evaluador representa a la dirección; habla su idioma.

  • 1

    Contexto y Urgencia (por qué importa AHORA)

    1-2 slides. Caso real del sector + costo de no actuar. "Una empresa como la nuestra en nuestro sector perdió X por no tener Y."

  • 2

    Diagnóstico (dónde estamos)

    2-3 slides. Inventario de activos + top 5 riesgos con ALE. Nivel de madurez actual. "Estamos en nivel 1.5 de 5."

  • 3

    Estrategia (qué proponemos)

    3-4 slides. Controles prioritarios con costo, impacto en riesgo y ROI. Quick wins vs. proyectos de largo plazo.

  • 4

    Plan de Acción (cómo llegamos)

    2-3 slides. Roadmap a 6-18 meses. Responsables, recursos, métricas de éxito.

  • 5

    Solicitud y Próximos Pasos

    1 slide. Qué necesitas de la dirección: presupuesto, autorización, apoyo político.

Preguntas Difíciles del Panel: Prepárate

❓ PREGUNTA 1 — Click para ver cómo responder

"Tu propuesta cuesta $500K. ¿Puedes garantizar que no vamos a sufrir un ataque si invertimos ese dinero?"

Respuesta correcta: "No puedo garantizar cero incidentes — ningún experto puede. Lo que puedo garantizar es que este plan reduce nuestro riesgo de pérdida anual esperada de $2.1M a $400K según nuestro análisis, y que cumpliremos con los requisitos regulatorios que actualmente nos exponen a multas de hasta $X. Es como el seguro de un edificio: no garantiza que nunca habrá incendio, pero sí que el impacto financiero será manejable."
❓ PREGUNTA 2 — Click para ver cómo responder

"¿Por qué comenzar con ISO 27001 y no simplemente comprar un buen antivirus y firewall?"

Respuesta correcta: "El antivirus y el firewall son controles técnicos que resuelven síntomas sin diagnosticar la enfermedad. ISO 27001 es el sistema de gestión que identifica sistemáticamente qué activos proteger, qué riesgos existen y qué controles son los más efectivos para NUESTRA organización específica. Sin esa base, compramos herramientas que quizás no cubren nuestros riesgos reales. Es la diferencia entre comprar medicamentos al azar vs. ir al médico primero."
❓ PREGUNTA 3 — Click para ver cómo responder

"¿Qué haría PRIMERO si mañana tuviéramos un presupuesto de solo $50,000 para seguridad?"

Respuesta correcta: Priorizar los CIS Controls de IG1 (básico): (1) Inventario de activos ($0, Excel); (2) MFA para todos los usuarios críticos (~$5K); (3) Gestión de parches (~$10K herramienta + proceso); (4) Segmentación básica de red ($15K firewall); (5) Backup offline verificado ($10K); (6) Capacitación de phishing ($5K). Estos controles mitigan el 80% de los vectores de ataque comunes por el menor costo.

Sesión 25 — Presentación Final del Proyecto Integrador y Cierre

💡 ESTRUCTURA DE LA DEFENSA

Cada equipo dispone de 15 a 20 minutos de presentación seguidos de 10 a 15 minutos de preguntas del panel evaluador. El panel valora la profundidad del análisis, la coherencia de las recomendaciones y la seguridad con que el presentador defiende sus decisiones técnicas y estratégicas.

Rúbrica de Evaluación

  • Análisis de riesgos (25%): metodología, datos, priorización
  • Controles propuestos (25%): pertinencia, costo-beneficio, factibilidad
  • Plan BCP/DRP (15%): BIA, RTO/RPO, estrategia de recuperación
  • Presentación ejecutiva (20%): claridad, narrativa, lenguaje de negocio
  • Defensa oral (15%): respuestas, manejo de preguntas, seguridad

Entregables Obligatorios

  • Inventario y clasificación de activos (ISO 27001)
  • Análisis de riesgos (cualitativo + cuantitativo ALE)
  • Plan de tratamiento de riesgos
  • Política de seguridad de la información
  • Plan de respuesta a incidentes (NIST 800-61)
  • BCP con análisis BIA
  • Dashboard de métricas (Power BI/Excel)
  • Presentación ejecutiva (15-20 slides)

Certificaciones para Tu Siguiente Paso

CompTIA Security+

Certificación de entrada más reconocida. Cubre fundamentos de todos los dominios del seminario. Examen: 90 preguntas, 90 min, $404 USD. Requisito en muchas empresas para roles de seguridad.

CEH (Certified Ethical Hacker)

EC-Council. Orientada a pentesting y evaluación de vulnerabilidades. Requiere 2 años de experiencia o capacitación oficial. Costo: $1,199 USD. Muy valorada en sector privado México.

CISM (Certified IS Manager)

ISACA. Para roles de gestión y gobierno (CISO). Requiere 5 años de experiencia. Costo: $575-760 USD. Complementa perfectamente los contenidos del Bloque IV del seminario.

CISSP

(ISC)². La certificación de mayor prestigio global. Requiere 5 años de experiencia en 2+ dominios. Costo: $749 USD. El estándar oro para CISO y roles senior de seguridad.

AWS Security Specialty

Amazon Web Services. Dominio avanzado de seguridad cloud en AWS. Requiere 5 años de IT + 2 en seguridad. Costo: $300 USD. Muy demandada por crecimiento de cloud en LatAm.

ISO 27001 Lead Auditor

PECB o Bureau Veritas. Para liderar implementaciones y auditorías ISO 27001. Costo: ~$1,500 USD (con examen). Habilitante para rol de consultor independiente o CISO.
