Hay 6,900 millones de smartphones en el mundo. Aprender desarrollo Android es como aprender a construir tiendas en el centro comercial más concurrido del planeta — todos están ahí.

1.1 FUNDAMENTOS TÉCNICOS
Anatomía de un Smartphone
Un smartphone es una computadora, una cámara, un GPS y un laboratorio de sensores en un bolsillo

🔮 Metáfora

Un smartphone es una computadora de escritorio miniaturizada que además tiene ojos (cámara), oídos (micrófono), sentido del equilibrio (acelerómetro) y siempre sabe dónde está (GPS). Como desarrollador, tienes acceso a todo ese hardware desde tu código.

📖 Procesamiento

  • SoC (System on Chip): CPU + GPU integrados
  • RAM: 4–16 GB (apps en memoria)
  • Almacenamiento interno: UFS 3.1/4.0
  • Chip de seguridad (TrustZone)

📖 Sensores clave

  • GPS / GNSS → geolocalización
  • Acelerómetro → detección de movimiento
  • Giroscopio → orientación
  • Cámara frontal y trasera
  • Sensor de huella / Face ID

📖 Conectividad

  • 4G LTE / 5G
  • WiFi 6/6E
  • Bluetooth 5.x
  • NFC (pagos sin contacto)
  • USB-C / carga inalámbrica

1.3 AMBIENTE DE DESARROLLO
Configurando Android Studio
El taller completo del desarrollador Android

🔮 Metáfora

Android Studio es el taller del carpintero: sin las herramientas correctas (SDK, emulador, Gradle) no puedes construir nada funcional, sin importar cuánto talento tengas. Configurarlo bien desde el inicio ahorra horas de frustración.

🔨 Pasos de instalación

  • 1. Descargar Android Studio Meerkat (2025)
  • 2. Instalar SDK Platform 35 (Android 15)
  • 3. Crear AVD: Pixel 8, API 35
  • 4. Habilitar HAXM o Hypervisor (emulador rápido)
  • 5. Instalar plugin Kotlin (ya incluido)

📖 Componentes clave

  • SDK Manager: descarga versiones de Android
  • AVD Manager: crea y gestiona emuladores
  • Gradle: sistema de construcción (el "chef" que ensambla tu APK)
  • Layout Editor: diseña vistas visualmente
  • Logcat: consola de depuración

2.1 KOTLIN — TIPOS Y VARIABLES
Kotlin: El lenguaje de Android moderno
Conciso, seguro, expresivo — todo lo que Java debió haber sido

🔮 Metáfora: val vs var

val es como grabar en piedra — una vez definido, no cambia. var es como escribir en pizarrón — puedes borrarlo y cambiarlo. La regla de oro: usa val siempre que puedas, var solo cuando necesites cambiar el valor.

// Variables e inferencia de tipos
val nombreApp: String = "CityReport"   // inmutable
var contadorReportes: Int = 0           // mutable
val version = 1.5                        // Double por inferencia
var activo: Boolean = true

// Null Safety — la protección contra NPE
var usuario: String? = null            // nullable con ?
val longitud = usuario?.length ?: 0   // Elvis: si null → 0
val upper = usuario!!.uppercase()      // !! = "confío en que no es null"

// String templates
val msg = "La app $nombreApp tiene ${contadorReportes + 1} reportes"

📖 Tipos básicos

  • Int — entero (32 bits)
  • Long — entero grande
  • Double — decimal
  • String — texto
  • Boolean — true/false
  • List, MutableList — colecciones

⚡ Kotlin vs Java

Kotlin elimina el 90% de los NullPointerExceptions que plagaban apps Android escritas en Java. Google reportó una reducción de 20% en crashes en apps que migraron a Kotlin.

2.1.4 ORIENTACIÓN A OBJETOS
OOP en Kotlin: Clases, Data Classes y Lambdas
Modelando el mundo real con objetos — el corazón de tu app

🔮 Metáfora: Data Class

Una data class es un formulario prellenado: automáticamente te da toString(), equals(), copy() y hashCode(). Sin data class tendrías que escribir 40 líneas de código para lo que Kotlin hace en 1.

// Data class — el modelo de datos de CityReport
data class Incidencia(
    val id: Int,
    val titulo: String,
    val descripcion: String,
    val latitud: Double = 0.0,
    val longitud: Double = 0.0,
    val resuelta: Boolean = false
)

// Enum para categorías
enum class Categoria { VIALIDAD, AGUA, LUZ, BASURA }

// Funciones de orden superior y lambdas
val incidencias = listOf(
    Incidencia(1, "Bache en Av. IPN", "Profundo y peligroso"),
    Incidencia(2, "Alumbrado roto", "Sin luz desde el lunes", resuelta = true)
)

val pendientes = incidencias
    .filter { !it.resuelta }          // filtra las no resueltas
    .map { it.titulo }               // extrae solo el título
    .sortedBy { it }                 // ordena alfabéticamente

when (incidencia.categoria) {

🔮 Extension Functions

Son como ponerle un accesorio nuevo a algo que ya existe — puedes agregar métodos a String, List o cualquier clase de Android sin modificar el código original. Es como personalizar tu teléfono con fundas sin romper el teléfono.

2.1.4 OOP AVANZADO
Interfaces, Herencia y Sealed Classes
Diseñando código flexible y mantenible

🔮 Metáfora: Interfaz

Una interfaz es un contrato laboral: no dice cómo hacer el trabajo, solo qué trabajo debe hacerse. Cualquier clase que firme el contrato promete implementar esos métodos, pero cada una puede hacerlo a su manera.

// Interfaz — define el contrato
interface IncidenciaRepository {
    suspend fun obtenerTodas(): List<Incidencia>
    suspend fun guardar(inc: Incidencia)
    suspend fun eliminar(id: Int)
}

// Sealed class — estados finitos y conocidos
sealed class EstadoUI {
    object Cargando : EstadoUI()
    data class Exito(val datos: List<Incidencia>) : EstadoUI()
    data class Error(val mensaje: String) : EstadoUI()
}

// Uso con when — el compilador sabe todos los casos
when (estado) {
    is EstadoUI.Cargando -> mostrarSpinner(true)
    is EstadoUI.Exito   -> mostrarLista(estado.datos)
    is EstadoUI.Error   -> mostrarError(estado.mensaje)
}

⚡ ¿Por qué Sealed Classes?

Representa estados de tu app que son finitos y conocidos. El compilador te obliga a manejar todos los casos en el when — si agregas un nuevo estado y olvidas manejarlo, el compilador te lo dice antes de ejecutar.

2.2 PATRÓN MVC
Model–View–Controller en Android
Separar responsabilidades: el principio más importante del diseño de software

🔮 Metáfora: El Restaurante

Model = La cocina (prepara los datos, sin saber cómo se presentan) · View = El mesero y la mesa (presenta la información al cliente) · Controller = El gerente (recibe la orden, coordina cocina y mesero). Sin esta separación, el chef atendería mesas y el mesero cocinaría — caos total.

// MODEL — solo datos
data class Incidencia(val id: Int, val titulo: String)

// CONTROLLER (Activity) — coordina
class ListaActivity : AppCompatActivity() {
    private val repo = IncidenciaRepositoryImpl()

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        binding = ActivityListaBinding.inflate(layoutInflater)
        setContentView(binding.root)
        cargarDatos()
    }
    private fun cargarDatos() {
        val lista = repo.obtenerTodas()
        binding.recyclerView.adapter = IncidenciaAdapter(lista)
        // VIEW se actualiza a través del adapter
    }
}
