2.3 PATRÓN MVP
Model–View–Presenter
La evolución de MVC: testing más fácil, lógica desacoplada

🔮 Metáfora: El Director de Orquesta

En MVP el Presenter es el director de orquesta: conoce la partitura completa (lógica de negocio) pero no toca ningún instrumento (no accede directamente a la UI). La View son los músicos que ejecutan — solo siguen instrucciones. El Model es la partitura (datos).

2.3 + 2.4 IMPLEMENTANDO MVP
Contratos de Interfaz en Kotlin
El patrón que hace tu código testeable y mantenible
// CONTRATO — la "constitución" del módulo
interface IncidenciaContract {
    interface View {
        fun mostrarIncidencias(lista: List<Incidencia>)
        fun mostrarError(msg: String)
        fun mostrarCargando(show: Boolean)
        fun navegarADetalle(id: Int)
    }
    interface Presenter {
        fun cargarIncidencias()
        fun onIncidenciaSeleccionada(id: Int)
        fun onViewDestroyed()
    }
}

// PRESENTER — sin imports de Android
class IncidenciaPresenter(
    private var view: IncidenciaContract.View?,
    private val repo: IncidenciaRepository
) : IncidenciaContract.Presenter {

    override fun cargarIncidencias() {
        view?.mostrarCargando(true)
        try {
            val lista = repo.obtenerTodas()
            view?.mostrarCargando(false)
            view?.mostrarIncidencias(lista)
        } catch (e: Exception) {
            view?.mostrarError(e.message ?: "Error desconocido")
        }
    }
    override fun onViewDestroyed() { view = null } // evitar memory leak
}

// FRAGMENT implementa la View del contrato
class IncidenciaFragment : Fragment(), IncidenciaContract.View {
    private val presenter = IncidenciaPresenter(this, RepoImpl())
    override fun mostrarIncidencias(lista: List<Incidencia>) {
        adapter.submitList(lista)
    }
}
3.1 CONTENEDORES Y CONTROLES
ConstraintLayout y ViewBinding
Construir interfaces responsivas y acceder a vistas sin casting

🔮 Metáfora: ConstraintLayout

ConstraintLayout es como armar muebles IKEA: cada pieza (View) se conecta a otra con restricciones (constraints). Si mueves una pieza, las que dependen de ella se ajustan automáticamente. Sin constraints = piezas flotando en el espacio.

<!-- activity_main.xml -->
<ConstraintLayout>
  <TextView
    android:id="@+id/tvTitulo"
    app:layout_constraintTop_toTopOf="parent"
    app:layout_constraintStart_toStartOf="parent"
    android:text="@string/app_name"/>
  <Button
    android:id="@+id/btnReportar"
    app:layout_constraintTop_toBottomOf="@id/tvTitulo"
    app:layout_constraintEnd_toEndOf="parent"/>
</ConstraintLayout>
// ViewBinding — sin findViewById
private lateinit var binding: ActivityMainBinding

override fun onCreate(...) {
    binding = ActivityMainBinding
        .inflate(layoutInflater)
    setContentView(binding.root)

    // Acceso directo y type-safe
    binding.tvTitulo.text = "CityReport"
    binding.btnReportar.setOnClickListener {
        presenter.onNuevoReporte()
    }
}

⚡ ¿Por qué ViewBinding?

Elimina NullPointerExceptions al acceder a Views. El compilador genera la clase binding automáticamente — si el ID no existe en el XML, el código no compila.

3.1 RECYCLERVIEW AVANZADO
RecyclerView con ListAdapter y DiffUtil
La lista más eficiente de Android — solo actualiza lo que cambió

🔮 Metáfora: DiffUtil

DiffUtil es como un editor de texto inteligente: en lugar de borrar toda la página y reescribirla cuando cambias una palabra, solo modifica exactamente la línea que cambió. Resultado: animaciones suaves y rendimiento óptimo.

// ListAdapter con DiffUtil automático
class IncidenciaAdapter(
    private val onClick: (Incidencia) -> Unit
) : ListAdapter<Incidencia, IncidenciaAdapter.ViewHolder>(DiffCallback()) {

    inner class ViewHolder(private val binding: ItemIncidenciaBinding)
        : RecyclerView.ViewHolder(binding.root) {
        fun bind(item: Incidencia) {
            binding.tvTitulo.text = item.titulo
            binding.tvCategoria.text = item.categoria.name
            binding.root.setOnClickListener { onClick(item) }
        }
    }

    override fun onCreateViewHolder(parent: ViewGroup, viewType: Int) =
        ViewHolder(ItemIncidenciaBinding.inflate(
            LayoutInflater.from(parent.context), parent, false))

    override fun onBindViewHolder(holder: ViewHolder, pos: Int) =
        holder.bind(getItem(pos))

    class DiffCallback : DiffUtil.ItemCallback<Incidencia>() {
        override fun areItemsTheSame(old: Incidencia, new: Incidencia) = old.id == new.id
        override fun areContentsTheSame(old: Incidencia, new: Incidencia) = old == new
    }
}

3.1.2 CAMERAX
Captura de Foto con CameraX
La API de cámara oficial de Jetpack — consistente en todos los dispositivos

🔮 Metáfora

CameraX es como contratar un fotógrafo profesional: le dices "quiero una foto aquí" y él se encarga de todo — enfocar, ajustar exposición, manejar el ciclo de vida. La API antigua de Cámara era como hacer todo tú mismo sin experiencia.

// CameraX — Preview + Captura de imagen
private lateinit var imageCapture: ImageCapture

private fun iniciarCamara() {
    val cameraProviderFuture = ProcessCameraProvider.getInstance(this)
    cameraProviderFuture.addListener({
        val cameraProvider = cameraProviderFuture.get()
        val preview = Preview.Builder().build().also {
            it.setSurfaceProvider(binding.viewFinder.surfaceProvider)
        }
        imageCapture = ImageCapture.Builder()
            .setTargetRotation(binding.root.display.rotation)
            .build()
        cameraProvider.bindToLifecycle(
            this, CameraSelector.DEFAULT_BACK_CAMERA,
            preview, imageCapture
        )
    }, ContextCompat.getMainExecutor(this))
}

private fun tomarFoto() {
    val archivoFoto = File(filesDir, "incidencia_${System.currentTimeMillis()}.jpg")
    val outputOptions = ImageCapture.OutputFileOptions.Builder(archivoFoto).build()
    imageCapture.takePicture(outputOptions, executor,
        object : ImageCapture.OnImageSavedCallback {
            override fun onImageSaved(output: OutputFileResults) {
                presenter.onFotoCapturada(archivoFoto.toUri())
            }
            override fun onError(exc: ImageCaptureException) {
                view?.mostrarError("Error al capturar: ${exc.message}")
            }
        })
}
3.1.2 GEOLOCALIZACIÓN
GPS con FusedLocationProvider
Obtener la ubicación del usuario de forma eficiente y precisa

🔮 Metáfora

FusedLocationProvider es como un detective experto en localización: combina GPS satelital, triangulación WiFi y torres celulares para darte la mejor ubicación posible con el menor consumo de batería. Sin él, tendrías que elegir solo uno de esos métodos.

// FusedLocationProvider — obtener ubicación actual
private val fusedLocationClient: FusedLocationProviderClient by lazy {
    LocationServices.getFusedLocationProviderClient(this)
}

fun obtenerUbicacion() {
    fusedLocationClient.lastLocation.addOnSuccessListener { location ->
        location?.let {
            presenter.onUbicacionObtenida(it.latitude, it.longitude)
        } ?: solicitarUbicacionForzada() // si lastLocation es null
    }
}

// Solicitar actualizaciones continuas
val locationRequest = LocationRequest.create().apply {
    interval = 10_000          // cada 10 segundos
    fastestInterval = 5_000   // mínimo 5 segundos
    priority = PRIORITY_HIGH_ACCURACY
}

fusedLocationClient.requestLocationUpdates(
    locationRequest,
    locationCallback,     // callback con nuevas coords
    Looper.mainLooper
)

// SIEMPRE detener en onStop para ahorrar batería
override fun onStop() {
    super.onStop()
    fusedLocationClient.removeLocationUpdates(locationCallback)
}
3.2 NOTIFICACIONES Y MENSAJES
Sistema de Notificaciones en Android
Canales, prioridades y notificaciones interactivas

🔮 Metáfora: Canales de Notificación

Los canales (obligatorios desde Android 8) son como los diferentes timbres de tu casa: el timbre principal (urgente, sonido alto), el timbre de servicio (silencioso, solo ícono), el de paquetería (badge sin sonido). El usuario decide qué canales activa — tú no puedes forzarlo.

// 1. Crear el canal (una sola vez, al iniciar la app)
fun crearCanalNotificaciones(context: Context) {
    if (Build.VERSION.SDK_INT >= Build.VERSION_CODES.O) {
        val canal = NotificationChannel(
            "canal_reportes",
            "Reportes CityReport",
            NotificationManager.IMPORTANCE_DEFAULT
        ).apply {
            description = "Avisos sobre nuevos reportes ciudadanos"
            enableLights(true); lightColor = Color.GREEN
        }
        context.getSystemService(NotificationManager::class.java)
            ?.createNotificationChannel(canal)
    }
}

// 2. Construir y mostrar notificación
fun notificarNuevoReporte(titulo: String, context: Context) {
    val pendingIntent = NavDeepLinkBuilder(context)
        .setGraph(R.navigation.nav_graph)
        .setDestination(R.id.listaFragment)
        .createPendingIntent()

    val notif = NotificationCompat.Builder(context, "canal_reportes")
        .setSmallIcon(R.drawable.ic_notification)
        .setContentTitle("Nuevo reporte: $titulo")
        .setContentText("Toca para ver los detalles")
        .setPriority(NotificationCompat.PRIORITY_DEFAULT)
        .setContentIntent(pendingIntent)
        .setAutoCancel(true)
        .build()
    NotificationManagerCompat.from(context).notify(1001, notif)
}
3.3 CREACIÓN Y MANEJO DE EVENTOS
Listeners, Coroutines y Flows
Eventos síncronos y asíncronos — el sistema nervioso de tu app

🔮 Metáfora: Listener vs Flow

Un Listener es como un vigilante: espera pacientemente hasta que algo ocurre y actúa. Un Flow es como un río de datos: fluye continuamente y puedes transformarlo, filtrarlo o combinar múltiples ríos en uno. Para búsquedas en tiempo real, el río es mejor que el vigilante.

// Listener tradicional — click simple
binding.btnEnviar.setOnClickListener { _ ->
    val titulo = binding.etTitulo.text.toString()
    if (titulo.isNotBlank()) presenter.enviarReporte(titulo)
    else binding.etTitulo.error = "El título no puede estar vacío"
}

// Búsqueda reactiva con doOnTextChanged + Coroutines
binding.etBuscar.doOnTextChanged { text, _, _, _ ->
    lifecycleScope.launch {
        presenter.buscar(text?.toString() ?: "")
    }
}

// Coroutine para operaciones de red/base de datos
lifecycleScope.launch {
    presenter.estadoFlow.collect { estado ->
        when (estado) {
            is EstadoUI.Cargando -> binding.progressBar.visibility = View.VISIBLE
            is EstadoUI.Exito   -> {
                binding.progressBar.visibility = View.GONE
                adapter.submitList(estado.datos)
            }
            is EstadoUI.Error   -> Snackbar.make(binding.root,
                estado.mensaje, Snackbar.LENGTH_LONG).show()
        }
    }
}
3.4 TRANSICIÓN ENTRE VISTAS
Navigation Component y Animaciones
El sistema de navegación oficial de Jetpack — un solo backstask, muchos destinos

🔮 Metáfora: El Metro

El NavGraph es el plano del metro: cada estación (Fragment) tiene un nombre único, cada línea (action) conecta estaciones con dirección específica, y el conductor (NavController) sabe exactamente dónde estás y cómo llegar a destino — incluyendo el botón Atrás.

// Navegar con Safe Args
val accion = ListaFragmentDirections
    .actionListaToDetalle(
        incidenciaId = item.id,
        titulo = item.titulo

findNavController().navigate(accion)

// Recibir en destino
val args: DetalleFragmentArgs by navArgs()
val id = args.incidenciaId
