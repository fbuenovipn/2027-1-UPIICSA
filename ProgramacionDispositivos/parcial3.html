4.1 PREFERENCIAS DE USUARIO
DataStore — Reemplazando SharedPreferences
Almacenamiento de preferencias asíncrono, seguro y reactive

🔮 Metáfora

DataStore es el cajón de tu escritorio: guarda las preferencias personales del usuario (tema oscuro, idioma, notificaciones activadas) que persisten entre sesiones. SharedPreferences era el cajón viejo que podía trabarse (bloqueaba el hilo principal). DataStore es el cajón nuevo con rieles suaves — nunca se traba.

// Definir las claves
private val Context.dataStore: DataStore<Preferences>
    by preferencesDataStore(name = "ajustes_usuario")

val TEMA_OSCURO = booleanPreferencesKey("tema_oscuro")
val NOTIF_ACTIVAS = booleanPreferencesKey("notif")

// Guardar
suspend fun guardarTema(oscuro: Boolean) {
    context.dataStore.edit { prefs ->
        prefs[TEMA_OSCURO] = oscuro
    }
}
// Leer (Flow reactivo)
val temaFlow: Flow<Boolean> =
    context.dataStore.data.map { prefs ->
        prefs[TEMA_OSCURO] ?: false
    }

4.1.2 ROOM — DAO Y DATABASE
DAOs y Base de Datos
Queries como funciones Kotlin — Room genera el SQL por ti
// 2. @Dao — operaciones sobre la tabla
@Dao
interface IncidenciaDao {
    @Query("SELECT * FROM incidencias ORDER BY fechaMs DESC")
    fun obtenerTodas(): Flow<List<IncidenciaEntity>>  // reactivo

    @Query("SELECT * FROM incidencias WHERE sincronizada = 0")
    suspend fun obtenerPendientes(): List<IncidenciaEntity>

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insertar(inc: IncidenciaEntity): Long

    @Update
    suspend fun actualizar(inc: IncidenciaEntity)

    @Delete
    suspend fun eliminar(inc: IncidenciaEntity)
}

// 3. @Database — punto de entrada
@Database(entities = [IncidenciaEntity::class], version = 1, exportSchema = false)
abstract class CityReportDatabase : RoomDatabase() {
    abstract fun incidenciaDao(): IncidenciaDao

    companion object {
        @Volatile private var INSTANCE: CityReportDatabase? = null
        fun getInstance(context: Context): CityReportDatabase =
            INSTANCE ?: synchronized(this) {
                Room.databaseBuilder(context, CityReportDatabase::class.java, "cityreport.db")
                    .build().also { INSTANCE = it }
            }
    }
}

⚡ Flow reactivo desde Room

Cuando devuelves Flow<List<T>> desde el DAO, Room emite automáticamente una nueva lista cada vez que se modifica la tabla. La UI se actualiza sola — sin polling ni callbacks manuales.

4.2 SERVICIOS EN LA NUBE

Backend completo sin escribir servidores: base de datos, auth, storage y más

🔮 Metáfora

4.2.1 FIREBASE FIRESTORE
Base de Datos NoSQL en Tiempo Real
Documentos y colecciones — más flexible que tablas SQL, más potente que JSON plano

🔮 Metáfora: La Carpeta de Archivos

Firestore es como una enorme carpeta de archivos en la nube: cada carpeta (colección) contiene documentos (registros), y dentro de cada documento hay campos (datos). No necesitas definir la estructura antes de usarla — es como crear carpetas y archivos libremente, sin formularios de alta.

📖 Estructura Firestore

  doc_abc123/         ← documento
    titulo: "Bache"
    latitud: 19.432
    longitud: -99.133
    resuelta: false
    fotos/            ← subcolección

val db = Firebase.firestore

// Guardar documento
suspend fun guardarEnNube(inc: IncidenciaEntity) {
    db.collection("incidencias")
      .document(inc.id.toString())
      .set(inc)
      .await() // suspending - no bloquea
}

// Listener tiempo real
db.collection("incidencias")
  .whereEqualTo("resuelta", false)
  .addSnapshotListener { snap, err ->
      snap?.documents
        ?.mapNotNull { it.toObject<IncidenciaEntity>() }
        ?.let { presenter.onDatosActualizados(it) }
  }
4.2 + 4.1 PATRÓN REPOSITORY
Repository: Unificando Room y Firestore
Single source of truth — la UI no sabe si los datos vienen de local o nube

🔮 Metáfora: El Almacén de Distribución

El Repository es como un almacén de distribución inteligente: cuando llega un pedido (solicitud de datos), primero revisa si hay stock local (Room). Si no hay o está desactualizado, hace el pedido a fábrica (Firestore). La tienda (UI) no sabe ni le importa de dónde viene el producto.

class IncidenciaRepositoryImpl(
    private val dao: IncidenciaDao,
    private val firestore: FirebaseFirestore
) : IncidenciaRepository {

    // La UI observa SIEMPRE Room (offline-first)
    override val incidencias: Flow<List<IncidenciaEntity>>
        get() = dao.obtenerTodas()

    // Guardar localmente primero, luego sincronizar
    override suspend fun guardar(inc: IncidenciaEntity) {
        dao.insertar(inc) // local inmediato
        try {
            firestore.collection("incidencias")
                .document(inc.id.toString())
                .set(inc).await()
            dao.actualizar(inc.copy(sincronizada = true))
        } catch (e: Exception) {
            // Sin conexión: WorkManager reintentará después
            Log.w("Repo", "Pendiente de sync: ${e.message}")
        }
    }
}
4.2.3 SINCRONIZACIÓN DE DATOS
ViewModel y StateFlow
La capa que conecta Repository con UI — sobrevive rotaciones de pantalla

🔮 Metáfora: El Portavoz

ViewModel es el portavoz del equipo: mantiene el estado actual de los datos y lo comunica a quien lo pida (Fragment, Activity). Si el periodista (pantalla) se va y vuelve (rotación), el portavoz sigue ahí con la misma información — no la pierde.

class IncidenciaViewModel(
    private val repo: IncidenciaRepository
) : ViewModel() {

    // Estado de UI como StateFlow
    private val _estado = MutableStateFlow<EstadoUI>(EstadoUI.Cargando)
    val estado: StateFlow<EstadoUI> = _estado.asStateFlow()

    init {
        viewModelScope.launch {
            repo.incidencias.collect { lista ->
                _estado.value = EstadoUI.Exito(lista)
            }
        }
    }

    fun guardarIncidencia(inc: IncidenciaEntity) {
        viewModelScope.launch {
            _estado.value = EstadoUI.Cargando
            try { repo.guardar(inc) }
            catch(e: Exception) { _estado.value = EstadoUI.Error(e.message ?: "") }
        }
    }
}

// En Fragment — recolectar el estado
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.estado.collect { estado -> actualizarUI(estado) }
    }
}
4.2 FIREBASE AUTHENTICATION
Autenticación de Usuarios
Email, Google Sign-In y sesiones persistentes — sin servidor propio

🔮 Metáfora: El Portero del Club

Firebase Auth es el portero más confiable del mundo: verifica que seas quien dices ser (autenticación), recuerda tu cara para la próxima vez (sesión persistente), y si pierdes tu credencial (contraseña), te ayuda a recuperarla. Tú como desarrollador solo le dices "deja pasar a usuarios válidos".

val auth = Firebase.auth

// Registro con email/contraseña
suspend fun registrar(email: String, pass: String): FirebaseUser? {
    return try {
        auth.createUserWithEmailAndPassword(email, pass).await().user
    } catch (e: FirebaseAuthException) { null }
}

4.2.2 ALMACENAMIENTO MULTIMEDIA
Firebase Storage — Fotos e Imágenes
Subir, descargar y gestionar archivos binarios en la nube

🔮 Metáfora: La Bodega Digital

Firebase Storage es como una bodega de depósito digital: cada archivo tiene una dirección única (path), puedes subir cajas (archivos) de cualquier tamaño, y recuperarlas con su URL. La bodega puede ser pública (todos ven las fotos) o privada (solo el dueño autenticado).

val storage = Firebase.storage

// Subir foto de una incidencia
suspend fun subirFoto(uri: Uri, incidenciaId: Int): String {
    val uid = Firebase.auth.currentUser?.uid ?: throw Exception("No autenticado")
    val ref = storage.reference
        .child("fotos/$uid/$incidenciaId.jpg")
    ref.putFile(uri).await()
    return ref.downloadUrl.await().toString() // URL pública
}

// Cargar imagen con Coil (biblioteca de imágenes)
binding.ivFoto.load(incidencia.fotoUrl) {
    crossfade(true)

    error(R.drawable.ic_error)
    transformations(RoundedCornersTransformation(16f))

⚡ Optimización de imágenes

Comprime la foto antes de subir: usa Bitmap.compress(JPEG, 80, outputStream). Una foto de 4MB comprimida pesa ~200KB — 20x menos transferencia y costo en Firebase.

4.2.3 SINCRONIZACIÓN
WorkManager — Tareas en Segundo Plano
Garantía de ejecución: aunque el teléfono se reinicie, el trabajo se completa

🔮 Metáfora: El Asistente Fiel

WorkManager es el asistente más confiable: aunque cierres la app, aunque el teléfono se quede sin batería y reinicie, él recuerda exactamente qué trabajo quedó pendiente y lo completa cuando las condiciones son correctas (hay WiFi, la batería no está baja). No hay que recordarle — él solo sabe qué hacer.

// 1. Definir el Worker
class SyncWorker(ctx: Context, params: WorkerParameters)
    : CoroutineWorker(ctx, params) {

    override suspend fun doWork(): Result {
        return try {
            val db = CityReportDatabase.getInstance(applicationContext)
            val pendientes = db.incidenciaDao().obtenerPendientes()
            pendientes.forEach { inc ->
                Firebase.firestore.collection("incidencias")
                    .document(inc.id.toString()).set(inc).await()
                db.incidenciaDao().actualizar(inc.copy(sincronizada = true))
            }
            Result.success()
        } catch (e: Exception) {
            Result.retry() // reintenta automáticamente
        }
    }
}

// 2. Programar ejecución periódica con restricciones
val syncRequest = PeriodicWorkRequestBuilder<SyncWorker>(
    15, TimeUnit.MINUTES
).setConstraints(
    Constraints.Builder()
        .setRequiredNetworkType(NetworkType.CONNECTED)
        .setRequiresBatteryNotLow(true)
        .build()
).build()

WorkManager.getInstance(context)
    .enqueueUniquePeriodicWork("sync_cityreport",
        ExistingPeriodicWorkPolicy.KEEP, syncRequest)
4.3 ARQUITECTURA FINAL
Clean Architecture: UI → Domain → Data
La arquitectura recomendada por Google para apps profesionales

🔮 Metáfora: El Edificio

La arquitectura en capas es como un edificio de 3 pisos: los cimientos (Data Layer: Room + Firestore) soportan la estructura (Domain Layer: repositorios y casos de uso), que sostiene el exterior (UI Layer: Fragments + ViewModels). Cada piso solo habla con el de abajo — nunca salta pisos.

📱
UI Layer — Fragments, Activities, ViewModels, StateFlow
↓ solicita · ↑ emite estado
⚙
Domain Layer — Repository Interface, Use Cases, Modelos de dominio
↓ solicita · ↑ retorna datos
🗄
Data Layer — Room DAO, Firestore, Firebase Storage, DataStore, WorkManager

⚡ Regla de dependencia

Las dependencias solo apuntan hacia adentro: UI conoce Domain, Domain conoce Data. Nunca al revés. Esto permite cambiar Room por otro ORM sin tocar la UI.
