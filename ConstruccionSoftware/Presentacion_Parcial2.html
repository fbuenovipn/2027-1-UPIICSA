Prototipo Funcional

Unidad II · 2.1 — De la interfaz al código
⚙️

📌 ¿Qué es un prototipo funcional?

Un prototipo funcional es software que ejecuta la lógica del negocio real, aunque aún no tenga la interfaz gráfica completa. El objetivo es validar que las reglas de negocio, cálculos y flujos de datos funcionan correctamente antes de invertir en la UI definitiva.

Metáfora: el motor del carro terminado y funcionando, aunque aún no tenga la carrocería pintada ni los asientos tapizados.

Mockup vs Prototipo Funcional

AspectoMockup (Parcial 1)Prototipo Funcional
¿Tiene código real?No (solo diseño)Sí (lógica funcionando)
¿Hace cálculos?NoSí
¿Guarda datos?NoSí (BD o archivo)
¿Tiene UI completa?Visual pero estáticaBásica o de consola
¿Se puede probar?Solo visualmenteSí, con datos reales
ObjetivoValidar diseño con clienteValidar lógica con equipo

Tipos de prototipos funcionales

🔄 Desechable (Throwaway)

Se construye rápido para validar una duda específica y luego se descarta. No es código de producción.

Ej: probar si el algoritmo de descuento funciona correctamente antes de integrarlo al sistema real.

🌱 Evolutivo (Evolutionary)

Se construye con calidad desde el inicio; irá creciendo hasta convertirse en el sistema final. El más usado en proyectos académicos.

Ej: el prototipo del Parcial 2 se convierte en el núcleo del sistema final del Parcial 3.

🎭 Horizontal vs Vertical

Horizontal: implementa todas las pantallas pero con funcionalidad mínima. Vertical: implementa un solo flujo completo de principio a fin con toda la funcionalidad.

Para el proyecto: prototipo vertical — el flujo de "registrar un pedido" completo.

💡 Sprint de desarrollo

En la Semana 6 (S6), el equipo codifica la lógica del negocio sin preocuparse por la UI. Una interfaz de consola o un menú de texto es suficiente para el prototipo funcional. La UI vendrá en el Parcial 3.

Diagrama de Clases UML

Unidad II · 2.2 — Estructura estática del sistema
📐

📌 ¿Qué modela el diagrama de clases?

Muestra la estructura estática del sistema: las clases que existen, sus atributos, sus métodos y las relaciones entre ellas. Es el diagrama UML más usado y el punto de partida para programar.

Sintaxis de una clase en UML

Pedido
ATRIBUTOS
- id: int
- fecha: Date
- total: double
- estado: String
MÉTODOS
+ calcularTotal(): double
+ cambiarEstado(e:String): void
+ aplicarDescuento(p:int): void
# validarItems(): boolean

Visibilidad: + público · - privado · # protegido · ~ paquete

Tipos de relaciones en UML

RelaciónSignificadoEjemplo
Asociación ——"usa / conoce"Cliente usa Pedido
Agregación ◇——"tiene, puede existir sin"Carrito tiene Productos
Composición ◆——"parte de, no existe sin"Pedido tiene Líneas (si se borra el pedido, se borran las líneas)
Herencia △——"es un tipo de"Administrador es un tipo de Usuario
Realización △- -"implementa interfaz"PedidoDAO implementa IRepositorio
Dependencia - -→"depende temporalmente"Reporte usa PedidoService (parámetro)

Quiz — Relaciones UML

En el sistema de la taquería, una Orden contiene ItemsOrden. Si se cancela la Orden, los Items también desaparecen. ¿Qué tipo de relación es?

✅ ¡Correcto! Es Composición: los ItemsOrden son parte constitutiva de la Orden. Si la Orden se elimina, los Items no tienen razón de existir por sí solos. El diamante relleno (◆) indica esta dependencia existencial.
❌ La relación correcta es Composición (◆). La Agregación (◇) se usa cuando la parte puede existir sin el todo. En este caso, los Items no tienen sentido sin su Orden.

Diagrama de Secuencia

Unidad II · 2.3 — Interacción entre objetos en el tiempo
↔️

📌 ¿Qué modela el diagrama de secuencia?

Muestra cómo interactúan los objetos del sistema a lo largo del tiempo para ejecutar un caso de uso específico. Complementa el diagrama de clases: las clases dicen QUÉ existe; la secuencia dice CÓMO colaboran.

Metáfora: la partitura de una orquesta — muestra qué instrumento (objeto) toca en qué momento y cómo se pasan la melodía entre sí.

💡 Tip para elaborar diagramas de secuencia

💡 Componentes vs Despliegue

Componentes: estructura lógica del software (qué módulos existen). Despliegue: estructura física (en qué máquinas corren). Para una MiPyME pequeña, el despliegue puede ser tan simple como "una sola PC con todo instalado".

Análisis de Componentes de la Arquitectura

Unidad II · 2.4 — Identificar, dividir y asignar
🔍

📌 ¿Qué es un componente en el desarrollo?

Un componente es una unidad de software cohesiva con responsabilidad bien definida, que puede desarrollarse, probarse y desplegarse de forma relativamente independiente. Es la unidad de trabajo de un miembro del equipo.

Criterios para identificar componentes

  • Alta cohesión: todo lo que está dentro del componente está relacionado con su responsabilidad central. Un componente de "Pedidos" no debe gestionar empleados.
  • Bajo acoplamiento: el componente depende lo menos posible de otros. Si cambio el componente de BD, no debería afectar la lógica de negocio.
  • Interfaz clara: el componente expone solo lo necesario al exterior (API interna), ocultando sus detalles internos.
  • Tamaño manejable: un componente que puede ser desarrollado por una persona en 1–2 semanas es el tamaño correcto para el proyecto.
💡 Asignación de componentes al equipo

En el Parcial 2, cada integrante del equipo es dueño de un componente: lo diseña, lo programa y lo prueba. La integración final (S9) une todos los componentes. Esto simula exactamente el trabajo en equipos reales de desarrollo de software.

Cohesión vs Acoplamiento

💪
Alta Cohesión

Todo dentro del componente tiene que ver con lo mismo. El componente hace UNA cosa y la hace bien.

✅ Meta siempre

🔗
Bajo Acoplamiento

Los componentes se conocen lo menos posible entre sí. Cambiar uno no rompe a los demás.

✅ Meta siempre

Metáfora: los departamentos de una empresa. Marketing y Contabilidad tienen alta cohesión (cada uno hace su cosa). Su bajo acoplamiento permite que Contabilidad cambie su sistema sin que Marketing deje de funcionar.

Desarrollo de Componentes

Unidad II · 2.5 — Buenas prácticas y principios SOLID
💻

Principios SOLID — El ABC del buen código

S — Single Responsibility Principle

Una clase tiene UNA sola razón para cambiar. PedidoService gestiona pedidos; no también envía emails.

Metáfora: el chef cocina, el mesero sirve. No el mismo.

O — Open/Closed Principle

Abierto para extender, cerrado para modificar. Agrega funcionalidad con nuevas clases, no editando las existentes.

L — Liskov Substitution Principle

Si B hereda de A, puedo usar B donde se espera A sin que el programa falle.

I — Interface Segregation Principle

Las interfaces deben ser específicas, no genéricas enormes. Mejor 3 interfaces pequeñas que 1 enorme.

D — Dependency Inversion Principle

Depende de abstracciones (interfaces), no de implementaciones concretas. Permite cambiar la BD sin tocar la lógica.

Ejemplo: Principio S y D en código

❌ Violación del Single Responsibility

class PedidoService {
  crearPedido() { ... }
  // ❌ No debería estar aquí:
  enviarEmailCliente() { ... }
  generarPDF() { ... }
}

✅ Con Single Responsibility

class PedidoService {
  crearPedido() { ... }
}
class EmailService {
  enviarConfirmacion() { ... }
}
class PDFService {
  generarComprobante() { ... }
}

💡 Buenas prácticas de codificación

  • Nombres claros y descriptivos: calcularTotal() no calc()
  • Funciones cortas: máximo 20–30 líneas por método
  • No repetir código (DRY: Don't Repeat Yourself)
  • Comentar el "por qué", no el "qué"
  • Usar control de versiones (Git) desde el primer día

Integración de Componentes

Unidad II · 2.6 — Integration Day: unir las piezas
🔗

📌 ¿Qué es la integración?

Es el proceso de unir todos los componentes desarrollados de forma separada para que trabajen juntos como un sistema cohesivo. La integración revela problemas de interfaces, contratos mal definidos y suposiciones incorrectas.

Metáfora: ensamblar el carro. Cada equipo fabricó su parte (motor, carrocería, ruedas) siguiendo especificaciones. La integración verifica que todas las piezas encajen correctamente.

Estrategias de integración

Big Bang

Todos los componentes se integran de una sola vez al final. Simple pero riesgoso: si falla, es difícil encontrar la causa.

⚠️ No recomendado para proyectos medianos/grandes.

Integración Incremental (Top-Down / Bottom-Up)

Top-Down: empieza desde la UI, simula las capas inferiores con stubs. Bottom-Up: empieza desde la capa de datos hacia arriba usando drivers de prueba. Se integra un componente a la vez y se prueba.

✅ Recomendado para el proyecto.

Integración Continua (CI)

Cada vez que un integrante hace commit, se ejecutan pruebas automáticas. Herramientas: GitHub Actions, Jenkins. Detecta problemas en minutos.

APIs Internas — El contrato entre componentes

Antes de que cada integrante desarrolle su componente, el equipo debe definir los contratos de las APIs internas: qué métodos ofrece cada componente, qué recibe y qué devuelve.

// Contrato del PedidoService (acordado antes de codificar)
interface IPedidoService {
  crearPedido(clienteId: int, items: Item[]): Pedido
  obtenerPedido(id: int): Pedido
  cancelarPedido(id: int): boolean
  calcularTotal(pedidoId: int): double
}

Con este contrato, el desarrollador de UI puede programar la interfaz aunque el PedidoService aún no esté terminado (usa un mock/stub).

Integration Day (S9) — Agenda sugerida

  • 0:00–0:30: Cada integrante presenta su componente terminado con pruebas unitarias pasando
  • 0:30–1:00: El Arquitecto integra los componentes siguiendo el orden definido (Bottom-Up)
  • 1:00–1:30: Pruebas de integración: ejecutar los flujos principales end-to-end
  • 1:30–2:00: Documentar bugs encontrados y asignar responsables. Comprometer fecha de fixes.

Verificación de Calidad del Software

Unidad II · 2.7 — Tipos de pruebas y defect tracking
✅

📌 Verificación vs Validación

Verificación: "¿Construimos el sistema correctamente?" — las pruebas técnicas comprueban que el código hace lo que el diseño especificó.
Validación: "¿Construimos el sistema correcto?" — el cliente confirma que el sistema resuelve su problema real.

Metáfora: verificación = el electricista revisa que los cables estén bien conectados según el plano. Validación = el dueño confirma que los enchufes están donde los necesita.

Pirámide de pruebas

E2E Tests
Pruebas de extremo a extremo
Pocas, lentas, costosas
Integration Tests
Pruebas de integración entre componentes
Medianas en cantidad
Unit Tests
Pruebas unitarias (método/clase)
Muchas, rápidas, baratas ← BASE

Regla: más pruebas unitarias, menos E2E. La base de la pirámide debe ser siempre la más grande.

Tipos de prueba explicados

🔬 Pruebas Unitarias

Prueban una sola función o método de forma aislada. Herramientas: JUnit (Java), pytest (Python), NUnit (.NET).

def test_calcular_total():
  pedido = Pedido()
  pedido.agregar_item("Taco", 15, 3)
  assert pedido.calcular_total() == 45

🔗 Pruebas de Integración

Prueban la interacción entre dos o más componentes reales (sin mocks). Verifican que los contratos de las APIs internas se cumplen correctamente.

🧪 Pruebas Funcionales

Prueban que el sistema cumple los requerimientos del negocio: "cuando registro un pedido de 3 tacos a $15 c/u, el total debe ser $45". Se derivan de los casos de uso.

🐛 Defect Tracking

Registrar todos los bugs encontrados en una tabla o herramienta: ID del bug, descripción, severidad (crítico/mayor/menor), responsable de arreglo, estado (abierto/cerrado). Herramientas: GitHub Issues, Jira, o una simple hoja de cálculo.

Herramientas CASE para UML

StarUML · draw.io · PlantUML — práctica en clase
🛠️

⭐ StarUML

Características

  • Aplicación de escritorio (Win/Mac/Linux)
  • Soporta todos los diagramas UML
  • Genera código esqueleto desde el diagrama
  • Interfaz drag-and-drop intuitiva
  • Versión gratuita funcional para estudiantes

Ideal para el proyecto

Diagrama de clases completo con atributos, métodos y relaciones. El "estándar" de la industria para modelado UML profesional.

Descarga: staruml.io

🔷 draw.io (diagrams.net)

Características

  • 100% gratuito y en línea
  • Sin instalación (navegador web)
  • Formas UML predefinidas
  • Colaboración en tiempo real
  • Integración con Google Drive y GitHub

Ideal para

Diagramas de componentes y despliegue. Colaboración de equipo en tiempo real (todos editan a la vez).

Acceso: app.diagrams.net

🌿 PlantUML

Características

  • Diagramas desde texto (código)
  • Control de versiones con Git (el diagrama ES código)
  • Integración con IDEs (VS Code, IntelliJ)
  • Gratuito y open source
  • Diagramas consistentes y reproducibles
